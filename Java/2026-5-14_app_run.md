## Spring Bootの設定ファイル
src/main/resources/application.properties
```
spring.application.name=demo
  spring.thymeleaf.cache=false
  spring.thymeleaf.prefix=classpath:/templates/
  spring.thymeleaf.suffix=.html
```

>設定の解説<br>
>- spring.application.name: アプリケーション名を指定します。<br>
>- spring.thymeleaf.cache=false: テンプレートキャッシュを無効化（開発時のみ）。<br>
>- spring.thymeleaf.prefix / suffix: テンプレートファイルのパスと拡張子を指定します。<br>

## `spring.thymeleaf.cache=false: テンプレートキャッシュを無効化（開発時のみ）`について

開発時に**HTML変更時に即反映されるようにする**

どういうこと？

↓

### まず「キャッシュ」って何？

一言でいうと、「一回作った結果を覚えておいて、次回は再利用する仕組み」

### イメージ

本来
```
毎回HTMLを読み込む
↓
毎回解析する
↓
毎回画面作る
```

は重い、だからSpringは

```
一回作ったHTML結果を保存
↓
次回は保存済みを使う
```

を行う。これがキャッシュ。

しかし、開発中は困る

例えば
```
<h1>Hello</h1>
```
を
```
<h1>Hello Spring</h1>
```
に変更したとする。

### cache=true の場合

Springが昔保存したHTMLを使う

だから、変更したのに画面変わらないことがある。

### cache=false の場合

`spring.thymeleaf.cache=false`にすると、毎回HTMLファイルを読み直すようになる。

だから
```
保存
↓
ブラウザ更新
↓
即反映
```
になる。

なぜ本番ではtrueが多いか？

**毎回読み直すと遅いから**

## `spring.thymeleaf.prefix / suffix: テンプレートファイルのパスと拡張子を指定します。`について

**HTMLテンプレートを探す場所**

どういうこと？？？？

↓

### 軽く流れ

Controllerの
```
return "hello";
```

は、「hello.html を表示してね」という意味ではあるんだけど、実際にはSpringが内部で

```
prefix + hello + suffix
```

を組み立てて探している

### 実際の設定

prefix
```
spring.thymeleaf.prefix=classpath:/templates/
```

**どこのフォルダを見るか**

suffix
```
spring.thymeleaf.suffix=.html
```

**どんな拡張子をつけるか**

### つまりSpring内部では

Controllerで、

```
return "hello";
```

を書くと、Springが内部で以下のものを作る

```
classpath:/templates/ + hello + .html
```

そして、

```
classpath:/templates/hello.html
```
を探す。

### classpathって何？

**Spring Bootが読み込む「特別な場所」**

今回だと

```
src/main/resources/
```

が classpath に含まれる、だから

```
classpath:/templates/
```

は実際には以下を指す

```
src/main/resources/templates/
```

### 具体的な実際の流れ

#### ① URLアクセス

```
GET /
```

#### ② Controller実行

```
@GetMapping("/")
public String helloForm(Model model) {
    return "hello";
}
```

#### ③ Springが受け取る

```
"hello"
```

#### ④ Thymeleafがテンプレートを探す

設定を使って

```
prefix
+
return値
+
suffix
```

つまり、

```
classpath:/templates/hello.html
```

#### ⑤ 実際のファイル

```
src/main/resources/templates/hello.html
```

が見つかる

#### ⑥ HTML表示

ブラウザへ返す

### なんでこんな仕組み？

**毎回フルパス書かなくていいから**

もし設定がなかったら以下のコードになってしまう

```
return "src/main/resources/templates/hello.html";
```

## アプリケーションを起動

```
./mvnw spring-boot:run
```

> このコマンドは、プロジェクトファイル直下（pom.xml があるディレクトリ）で実行してください。

これの意味**Mavenを使ってSpring Bootアプリを起動するコマンド**

### このコマンドについて

### ① `./mvnw`

これは **Maven Wrapper**

普通はMavenを使うには、PCにMavenをインストールする必要がある

しかし、Spring Initializrで作ったプロジェクトには、mvnw というファイルが入っている

これを使うと、**PCにMavenを直接インストールしていなくても、そのプロジェクト用のMavenを使って実行できる** というイメージ

Spring Initializrの説明でも、Maven Wrapperが自動で含まれるため、Mavenをインストールしなくてもプロジェクトをビルドできると説明されている。（https://docs.spring.io/initializr/docs/0.4.x/reference/html/getting-started.html?utm_source=chatgpt.com）

### ② `spring-boot:run`

これは、

**Spring Bootアプリを起動してね**

という命令(pom.xml に書かれている設定やDependenciesをもとに、アプリをビルドして起動)

## なぜ pom.xml がある場所で実行する?

**pom.xml とは？**

pom.xml は、Mavenプロジェクトの設定ファイル

ここに、

- プロジェクト名
- Javaのバージョン
- Dependencies
- Spring Bootの設定
- ビルド方法

などが書かれている

Mavenは pom.xml を見て動くので、例えばプロジェクトが以下のようになっているなら、

```
demo/
├── pom.xml
├── mvnw
├── mvnw.cmd
└── src/
```

この demo/ の中で実行する必要がある

```
cd demo
./mvnw spring-boot:run
```
